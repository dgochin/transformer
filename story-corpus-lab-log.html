Lab Log · Synthetic Data Pipeline

Teaching a machine to tell a coherent children's story

A build diary: generating a TinyStories-style training corpus, discovering why a local model kept breaking the laws of physics, and moving the work off a rented GPU onto an API that judged 220,000 stories for about eleven dollars.

Goal · graded story corpus for a small LM Levels · L1 "Toddler core" first Kept · 206k of 220k judged Cost · $40.02 pod + $10.90 API

1 The problem

Every story was individually fluent and collectively nonsense

The plan was ordinary: prompt a language model to write thousands of tiny, simple stories, then train a small model on them the way the original TinyStories paper did. The trouble showed up on the very first read-through.

The stories parsed. The grammar was fine. But the world inside them was broken in a way that only a careful human reader catches: a chicken sitting in a blanket fort, a table inside a cave, a cup used to "make something to eat." The model was pattern-completing plausible sentences without any model of how objects actually behave.

Cull · impossible physics "He stretched the hoop to make it smaller. He pulled as hard as he could, and the hoop got smaller and smaller."
Stretching makes things bigger. The sentence is fluent; the physics runs backwards.
LESSON A language model is not a world model. Telling it to "obey physics" adds a phrase it may ignore. It doesn't install the ability to check. You cannot prompt your way out of this class of error; you have to catch it afterwards.

2 The generator

Coherence by construction: the knobs we turned

Rather than let the model freely pair any noun with any action (which is how you get chickens in blanket forts), each story is assembled from a scenario group, a bundle of things that naturally belong together. Then several dials add variety without breaking sense.

Scenario groups

24 themed bundles (kitchen & baking, garden, pond, forest…). The noun, setting, and action for one story are all drawn from the same group, so incompatible pairings can't occur.

theme: "pond and stream" → { frog, reed, net… }

Nouns

The single object a story centres on. Tightened repeatedly to remove "make-able" nouns that lured the model into building things it shouldn't (a plate becoming bakeware, a cloth becoming a boat).

noun: "watering can"

Settings

Where it happens, bound to the theme so daylight trails don't end up needing lanterns, and cave stories don't hold furniture.

setting: "a flower garden"

Features & interactions

The one thing that happens. Level 1 features are solo-achievable; anything social ("shares food") was moved to two-character levels so a lone toddler wasn't forced to invent a second child.

feature: "waters a young tree that is still growing"

Verb & adjective injection

Optional part-of-speech seeds to nudge sentence variety at higher levels without dictating the plot.

verb: null · adjective: null (L1 kept plain)

Opening-style rotation

The biggest variety fix. Left alone, the model opened every story with the character's name. So a specific opening is drawn per story: a sound, a question, the weather, a line of speech, the time of day, or sometimes the name.

"One frosty morning…" · "Tap, tap went the rain…"

Diversity doesn't come from asking the model to be diverse. It comes from varying the input and letting the model render each variation faithfully.

LESSON Whatever freedom you give a model, it finds a rut inside it. "Vary the opening" → every story name-first. Add a rotation → every sound opening became "Tap, tap, tap" triples. Each rut is fixed the same way: name the variants explicitly and rotate them.

3 The judge

Generate fast, check slow

Scenario design removed whole classes of error at the source, but not all of them. A model handed a slightly awkward pairing still improvises broken physics to reconcile it. So a second model reads every finished story and answers one narrow question: does anything here break physics, logic, or its own premise?

The rubric grew to five error types, each anchored with a concrete example so the verdict is reproducible rather than vibes-based:

Error typeWhat it catchesexample
physicsAn action producing an impossible resulta stone floats unaided
settingAn action that makes no sense in the placepouring water underwater
contradictionTwo things that can't both be truenothing fell out, yet it was full
causeeffectA result that doesn't follow from its causea fort that reaches a high shelf
premiseA later line that breaks the story's own setupsun preserves a snowflake
Keep · coherent, simple, fine "Hugo watered his little pear tree every day. One day a pear grew, and he picked it and ate it."
Plain, even a little thin, but every step follows. The judge keeps simplicity; it only culls breaks.

Calibration mattered as much as the rubric. A judge that culls everything looks accurate on bad stories and is useless. The keeper of last resort was a firm rule to keep imaginative liberty (a child calling a pressed shape "her snowflake") and normal processes (dough rising), and cull only genuine breaks.

thinking mode off

LESSON When replacing a story is nearly free, an over-strict judge is the correct judge. Culling a good story costs a fraction of a cent to regenerate; keeping a bad one poisons the training data permanently. The asymmetry says: when in doubt, cull.

4 The pivot

From a local GPU to a $10 API pipeline

The first corpus was generated on a local qwen3:14b model running on a rented A40 pod, billed by the hour and babysat through disk limits, VRAM overflows and re-pulls. Testing the same prompts on the DeepSeek API changed the economics and the quality at once.

Two settings made or broke the switch. Thinking mode had to be forced off: a reasoning block per story would have multiplied the token cost tenfold. And the few-shot examples turned out to change story richness, not just rhythm, so they were kept for longer, more developed stories.

Then the numbers settled the model question outright. Judged on identical scenarios and prompts:

And the cost ran the same direction as the quality. The pod, rented by the hour and nursed through disk limits and re-pulls, cost four times as much as the API while producing the less reliable tier:

LESSON "I have free credits" pulled toward the pod; the real cost was elsewhere. The API generated cleaner stories and judged the entire corpus for a quarter of what the pod's raw generation cost, and removed the babysitting entirely. Free compute for the weaker tool is a false economy.

The corpus is a blend: roughly 130,000 stories generated on the local pod ($40.02) and 85,000 on the DeepSeek API, with all 220,000 judged by DeepSeek for a fraction of a cent each ($10.90 covering the API generation and the full judging pass). Of 220,241 stories judged, 13,610 were culled. The breakdown is a portrait of exactly the failure modes the whole pipeline was built to catch:

physics5,751
causeeffect5,556
premise1,094
contradiction895
setting314

Physics and cause-and-effect together are 83% of all breaks: the signature of a model welding two ill-fitting ideas together with an impossible mechanism. The 1,018 premise breaks are the subtle, distributed kind the local judge kept missing; catching them was the clearest argument for the API judge.

For comparison, the corpus this project imitates, the original TinyStories set, is roughly 2 million stories. This is about a tenth of that scale. And the cost split points the way forward: since the API tier was both cleaner and cheaper, every level from here on is generated on DeepSeek, and the pod stays off.

6 What carried over

Principles worth keeping

Stripped of the specifics, the project kept returning to the same few ideas, the ones that transfer to any work with these models.

Shape the input

Don't ask for diversity or coherence; build it into what you hand the model. Scenario groups and opening rotation did more than any instruction.

Verify the output

Don't trust an instruction to hold. A model told to obey physics still won't. Check afterwards with a separate, stronger reader.

Instructions aren't equal

Positive beats negative ("start with the weather" > "don't start with the name"). A strong prior beats a weak rule every time.

Let cost set the strategy

When regeneration is nearly free, cull hard and top up. Calibration only matters when the corpus is fixed or expensive.

The project started by debugging why a chicken was sitting in a blanket fort. It ended with a fast, cheap, coherence-filtered pipeline, and most of the distance between the two was learning where to stop trusting the model and start checking it.
